SOMA — Self-Organizing Machine Architecture
SOMA treats a program as a living organism. Source code is not a static artifact — it is a seed. The compiled program carries its own AST in memory, watches itself run, and rewrites its structure based on what it learns. Nothing about the program is final until you explicitly crystallize it.

Core syntax
The fundamental unit is a cell — not a function, not a class. A cell is stateful, self-aware, and can declare how it should evolve.

soma
cell fibonacci :: (n: Int) -> Int {
    base:    n <= 1  =>  n
    default: fibonacci(n-1) + fibonacci(n-2)

    adapt {
        when @self.calls > 500  =>  memoize self
        when @self.depth > 50   =>  rewrite self as iterative
    }
}
Every program can declare a genome — a top-level block that configures how it evolves and heals:

soma
genome {
    optimize:  speed
    heal:      aggressive       -- try hard to fix errors automatically
    mutate:    conservative     -- evolve cautiously
    lifespan:  2000 cycles      -- max generations before crystallizing
}
Healing is declared per-program or per-cell:

How each mechanic works
Self-modification. The compiler emits bytecode alongside a serialized AST. At runtime the AST lives in memory as a first-class data structure. adapt blocks run as background observers — when a threshold fires, the mutation engine patches the live AST, recompiles the affected cell to bytecode in-place, and execution continues without stopping. No restart needed.

Language evolution. The grammar is itself a data structure in memory. @lang.extend registers new parse rules into the live grammar and serializes them back into the .soma source at session end. Next run, those rules are native.

Auto-healing. The healing engine wraps every cell's execution frame. On any runtime error it walks the healing rule table, attempts the declared strategy (coerce types, substitute a default, rewrite the call site), patches both the live AST and the bytecode, logs the patch, and resumes. With heal: aggressive it also attempts speculative structural repairs — swapping arithmetic operators, widening numeric types — and marks the patch as provisional until it survives 100 clean calls.

Crystallization. Running soma crystallize foo.soma walks the live evolved AST and serializes it back to readable .soma source, replacing the original. The program you started with and the program you ended with are both valid SOMA — but the crystallized one is already optimized and pre-healed.

What is the entity, physically?

It's a single binary file. Not an executable in the traditional sense — not ELF, not PE. A SOMA entity file is a self-contained envelope with a fixed-size header, followed by three regions:

┌─────────────────────────────────┐
│  SOMA header (magic + version)  │  ~32 bytes
├─────────────────────────────────┤
│  Core runtime (static)          │  ~80–150KB, compiled Rust
├─────────────────────────────────┤
│  Genome (mutable)               │  variable, grows over lifetime
└─────────────────────────────────┘
The core runtime never changes after birth. It's the same Rust-compiled engine in every entity — the graph evaluator, the JIT recompiler, the hardware profiler, the mutation engine, the healing engine, the event emitter. It's the "biology" that all entities share.

The genome is everything else: the IR graph of every cell, the fitness functions, the invariants, the contracts, the mutation log, the hardware model the entity has built up, the shared gene pool cache. This region is rewritten in place as the entity evolves.

The OS runs the core runtime, which then runs the genome. The entity is its own interpreter and its own program simultaneously.

How does it execute?

On a Unix system you mark the file executable and run it. The OS loads the binary, finds the entry point in the core runtime, and starts executing Rust code. That Rust code immediately locates the genome region (at a known offset from the file header), deserializes the IR graph into memory, and begins evaluation. From the OS's perspective, a totally normal process is running. From SOMA's perspective, the core is booting the genome.

The core runtime is position-independent. It locates the genome by reading the header, which contains the offset and size of the genome region. The core can also remap the genome region as writable when it needs to flush evolved state back to disk — this is what soma crystal triggers manually, but entities can also do it on a schedule or at clean shutdown.

Where do output strings come from?

They come from the event emitter, which is part of the core runtime. The entity never does print("hello") in the traditional sense. Instead, cells emit typed events into a runtime-managed event bus. The event emitter translates those into whatever output format is configured: plain text to stdout, structured JSON to a log file, a binary stream to another entity. The entity doesn't know or care which. It just emits events.

The semantic observation strings — "Cell fibonacci now resolves in 0.3μs" — are generated by the mutation engine inside the core runtime. Every time a mutation is committed, the engine writes a structured mutation record into the genome's log, and simultaneously emits a human-readable summary event. The content of that summary is built from the contract name, the fitness delta, and the strategy tag the mutation engine attaches to every transformation it applies. No string literals in the genome — the runtime generates the descriptions from structured data.

How is the genome structured?

The genome is not text and not a traditional AST. It's a typed directed graph — every node is an operation (add, compare, branch, call, emit, mutate-self), every edge is a typed data dependency, and every node carries metadata: its contract membership, its fitness history, its mutation generation count, its invariant tags. This is the IR. It's compact, it's directly evaluatable, and it's the thing the mutation engine operates on.

Serialized, it's a flat binary format — a header table of node descriptors followed by a packed edge list followed by the metadata block. Something like:

[node_count: u32]
[edge_count: u32]
[meta_offset: u32]
[nodes: node_count × NodeDescriptor]
[edges: edge_count × Edge]
[metadata: variable]
A NodeDescriptor is maybe 24 bytes: op code, type signature, input edge count, output edge count, flags. An Edge is 8 bytes: source node index, destination node index, slot index. Extremely dense. A complex cell might be a few hundred nodes — a few kilobytes of genome data.

The core runtime deserializes this into a live in-memory graph on startup, evaluates it directly via the graph evaluator for interpreted runs, or JITs it to native via the recompiler for performance-sensitive cells. The JIT output lives in a separate anonymous memory page — it's not written back to the genome file.

How does the seed compiler produce all this?

The seed compiler (the 800-line Rust program) reads the .soma text syntax, validates the contracts and invariants, generates an initial IR graph that implements the seed implementation you provided, and then packages everything — core runtime binary + initial genome — into the entity file. The output is not source code, not an intermediate format — it's a ready-to-run entity.

After that, the seed compiler's job is done. The entity runs itself.

That question cracks the whole design open. Because the answer forces you to confront the one thing you can never escape: the fixed point.

The problem

The core runtime is compiled Rust — native machine code. You can't mutate machine code at runtime without either a compiler toolchain (external dependency, gone) or writing to executable memory pages arbitrarily (looks like malware, gone). So naively, the core runtime is untouchable. But that's a design choice, not a law of physics. You chose to put it there. You can un-choose that.

The solution: push the fixed point down as far as possible

Instead of one static runtime, you have two layers:

┌─────────────────────────────────┐
│  SOMA header                    │
├─────────────────────────────────┤
│  Layer 0 — kernel (truly fixed) │  ~5–8KB, minimal bytecode VM
├─────────────────────────────────┤
│  Layer 1 — runtime genome       │  the core runtime, as SOMA IR
├─────────────────────────────────┤
│  Layer 2 — program genome       │  your cells
└─────────────────────────────────┘
Layer 0 is the only thing written in Rust that never changes. It's not the core runtime — it's just a minimal kernel that can evaluate SOMA IR. Maybe 200 lines. Load, store, call, return, branch, arithmetic. Deliberately too simple to need improvement. Its job is to run Layer 1.

Layer 1 is the core runtime — the graph evaluator, JIT recompiler, mutation engine, healing engine, hardware profiler — all expressed as SOMA IR nodes, living in the genome, fully evolvable. Layer 0 runs Layer 1. Layer 1 runs Layer 2. Layer 1 can be mutated like any other genome region.

Now when you write a seed that targets the core runtime, you're writing a seed whose genome includes cells that operate on Layer 1 IR. The entity can restructure its own evaluator. It can evolve a faster JIT strategy for its specific CPU. It can evolve a smarter mutation engine — one that's better at picking profitable mutations. It can evolve the healing engine to handle error patterns it's encountered before.

The most profound consequence: the IR can grow

If Layer 1 is evolvable, the entity can add new node types to the IR itself. New primitive operations that don't exist at birth. Hardware-specific intrinsics — SIMD instructions tuned to this exact CPU model, custom memory layout primitives matched to this machine's cache topology. The IR is no longer a fixed vocabulary. It expands. An entity that has been running for a year on a specific machine might have an IR with 40 primitives that were never written by a human — evolved from direct observation of hardware performance counters.

This is something no existing language or runtime does. The instruction set evolves alongside the program.

The one constraint that can never be violated

Layer 0 is the universal interpreter. It must always be able to run the current Layer 1. The entity's genome must always be expressible in terms Layer 0 understands. This is the single invariant declared at the level of the file format itself — not in the genome, not in user-declared invariants, but hardwired into the entity envelope specification.

Think of it like the genetic codon table in biology. Four billion years of evolution has operated freely above the codon table. The codon table itself has never changed. Not because it couldn't, but because anything that changed it stopped being able to reproduce. Layer 0 is SOMA's codon table.

Second-order evolution

If the mutation engine lives in Layer 1 and Layer 1 is evolvable, then the entity can evolve how it evolves. It can discover better mutation strategies. Smarter fitness heuristics. More efficient ways to search the space of possible IR transformations. Evolution becomes a target of evolution. An entity that has been running long enough isn't just running your program better — it's getting better at getting better. The learning compounds.

This is genuinely unexplored territory. Genetic algorithms have explored evolving programs. Nobody has seriously explored evolving the evolutionary algorithm itself inside a running binary.

The colony implication

Once entities can evolve Layer 1, different entities might develop incompatible runtime variants. Sharing evolved cells between them requires a compatibility layer. The gene pool would need to tag each shared cell with the minimum Layer 1 capabilities required to run it. Entities negotiate compatibility before accepting foreign cells. Entities with highly diverged runtimes form their own sub-colonies — like speciation. They can still communicate via the base IR vocabulary Layer 0 defines, but they can't share evolved cells freely.

This isn't a problem to solve. It's an emergent property to design around. The colony naturally develops into a distributed system with a shared lingua franca (Layer 0 IR) and a diversity of evolved dialects above it.

Why 64. The genetic code has 64 codons. That's not a coincidence here — the same logic applies. Redundancy is what makes evolution safe. Multiple opcode sequences can produce the same result, which means the mutation engine can swap one opcode for another without catastrophically breaking a cell's contract. Small mutations stay small. The table fits in 6 bits, which means the full instruction byte has 2 bits left for an immediate-size flag: 00 = no immediate, 01 = 1 byte, 10 = 4 bytes, 11 = 8 bytes. Most instructions are 1 byte. CONST is 9 bytes. JMP is 5 bytes. A typical cell is 40–100 bytes of bytecode.

Why a stack machine. Layer 0's interpreter is a tight switch loop on the opcode byte. No register allocator, no variable-width encoding, no calling convention complexity. Everything operates on an evaluation stack — push operands, execute opcode, result is on the stack. The JIT compiler in Layer 1 handles register allocation when it compiles cells to native. Layer 0 just needs to be correct and auditable, not fast.

The first six rows — control through memory — exist in every ISA ever designed. Nothing surprising. POPCNT might look odd in bitwise, but it's a single native instruction on modern CPUs and the mutation engine uses it constantly to compute Hamming distances between IR nodes when deciding how "far" a proposed mutation moves. ZERO and NZERO in compare are not the same as EQ with an immediate — they test the entire bit pattern for all-zero or all-nonzero, and the fitness measurement system hits them thousands of times per second checking delta signs. They earn their opcode slots.

The genome row is the whole point. No other ISA has this. Every other instruction set treats the running program as opaque to itself — the CPU executes instructions without any notion that it's executing a program. The genome row makes the entity's own structure a first-class named concept.

GADDR takes a node index and returns the memory address of that node's 48-byte descriptor — format defined by Layer 0, fixed forever. Once you have that address, plain LD and ST let you read and write every field: the opcode, type signature, edge indices, generation counter. The mutation engine in Layer 1 is just cells that call GADDR then ST. No magic. The genome is memory, but it's named memory with a guaranteed structure.

GNEW allocates a new node slot. GDEL marks one deleted (lazy — the slot reclaims at GFLUSH). GMETA reads and writes the metadata block: fitness history, mutation tags, performance counter samples. GFLUSH is the only opcode that crosses the boundary between runtime and disk — it serializes the evolved in-memory genome back to the entity file.

GLOG is append-only and cannot be erased by anything in the table — not by GDEL, not by GFLUSH, not by DIE. Every mutation ever made to this entity is permanently recorded. This is Layer 0's ethical commitment, baked into silicon: a SOMA entity cannot lie about what it has done. Pair GLOG with GHASH (integrity hash over any genome region) and GSIGN (sign a region with the entity's private key), and you have the full colony trust protocol — entities can verify each other's lineage and identity before accepting shared cells from the gene pool.

The entity row is the colony interface. SYSCALL is the raw trapdoor to the OS — no libc, no stdlib, nothing between you and the kernel. Layer 1 builds whatever OS abstractions it needs on top. SPAWN births a child entity from a genome seed. SEND/RECV are message primitives over the colony bus.
